# v4.0.0

## Highlights

- New `PostgresQueue` adapter with auto-migrated `mail_time_queue` table, JSONB mail options, due/pending indexes, and `prefix` scoping.
- Upgraded to [`josk@^6`](https://github.com/veliovgroup/josk) — exactly-once across the cluster, with Mongo / Redis / Postgres scheduler adapters.
- Added lifecycle methods: `mailTime.ready()` and `mailTime.destroy()`.
- Default `josk.zombieTime` raised to **60000 ms** (was `32786`) to match JoSk best-practice minimum.
- New `josk.onError` hook routed to MailTime's logger by default.
- Removed the `deepmerge` runtime dependency — replaced by a tiny inline `deepMerge` tailored to nodemailer mail option shapes. Single runtime dep now: `josk`.
- Node engine bumped to `>=20.9.0`. Bun ≥ 1.1.0 supported.
- **New `mailTimePreset(name, overrides)` helper + `presets` / `presetNames` exports.** One-line setup for the common email shapes: `transactional`, `otp`, `newsletter`, `marketing`, `notifications`, `alerts`. The function deep-clones the named preset and deep-merges your overrides (scalars win, nested `josk` composes), returning a ready-to-pass `MailTime` constructor config — you supply `queue` / `transports` / `josk.adapter` / `prefix` on top. Built-in presets live in `presets.js` (frozen); the helper is re-exported from `mail-time`. See README §"Settings presets" for the full table. No behavioral change for hand-coded configs — purely additive.
- **New `isSending` per-row lock + bounded parallel sends inside a single MailTime instance.** Each row now carries `isSending` (the lock) and `sendingAt` (when the lock was taken). Claim updates set `{ isSending: true, sendingAt: now, tries: tries+1 }` atomically, guarded by `isSent=false AND isFailed=false AND isCancelled=false AND tries=task.tries AND (isSending=false OR sendingAt <= now - sendingTimeout)`. The storage CAS makes it impossible for two workers — in the same instance or across the cluster — to flip the same row at the same `tries`, so duplicate delivery is prevented even when sends run in parallel.
- **New `mode` option** (`'one' | 'batch'`, default `'batch'`). Mirrors JoSk's `execute`: `'batch'` claims every due row per tick; `'one'` claims a single row per tick (fairness across cluster nodes).
- **New `concurrency` option** (default `1`). Up to N parallel SMTPs per instance, dispatched through an internal worker pool. Increase to scale throughput on a single host before resorting to more `prefix`es or more pods.
- **New `sendingTimeout` option** (default `300000` ms / 5 min). Stale-lock recovery — a row stuck `isSending=true` because its worker died is reclaimable after this window.
- **New `drain()` method.** Resolves once every in-flight send finishes. Pair with `destroy()` for graceful shutdown.
- **JoSk lease released faster.** `___iterate` returns as soon as the scan + claim phase completes; SMTP work continues in the background pool. Other ticks (this node or others) can immediately pick up rows still `isSending=false`.
- **Per-recipient retries.** When a multi-`to` / `cc` / `bcc` send is partially rejected by the SMTP server, MailTime now retries **only** the un-accepted addresses on the next attempt — delivered recipients never receive a duplicate copy.
- `task.mailOptions[i].accepted` accumulates lowercased addresses confirmed by `info.accepted` across all attempts. Original `to` / `cc` / `bcc` lists on each `mailOption` are never mutated.
- `task.mailOptions[i].rejected` is populated as `{ address, error }` per un-delivered recipient when the retry budget (`retries` / `maxTries`) is exhausted; `onError(err, task, info)` then fires once with an aggregated `Recipients rejected after N attempts: <addresses>` error.
- `onSent` fires once on full delivery; `onError` fires once on retry exhaustion. Transports that don't populate `info.accepted` / `info.rejected` fall back to the existing "any-accepted = success / none-accepted = retry" behavior.

## Adapters

- **Redis** — `push` now uses `MULTI` when available for atomic write of letter + sendAt + concat keys. Iterate parses uuid via prefix-aware key splitting (handles prefixes containing colons).
- **Mongo** — index conflicts (code 85) handled cleanly by dropping the legacy index and recreating. Uses `collection.collectionName` for error messages.
- **Postgres** — `iterate` returns early when `mailTimeInstance` is missing instead of throwing. Schema setup consolidated into one idempotent `CREATE TABLE IF NOT EXISTS`.
- All three built-in adapters (Redis / Mongo / Postgres) now accept `iterate({ limit, sendingTimeout })` and skip rows whose `isSending=true` lock is still fresh.
- **Postgres** schema gains `is_sending BOOLEAN NOT NULL DEFAULT false` and `sending_at BIGINT NOT NULL DEFAULT 0`. Existing tables are auto-migrated via `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`. The due-row index now covers `(prefix, is_sent, is_failed, is_cancelled, is_sending, sending_at, send_at, tries)`.
- **Mongo** adapter updates its compound index to include `isSending` and `sendingAt`. Claim updates use `$or: [{ isSending: { $ne: true } }, { sendingAt: { $lte: now - sendingTimeout } }]`.
- **Redis** adapter serializes `isSending` and `sendingAt` inside the letter JSON. WATCH/MULTI claim path now consults both.
- Adapters dispatch each due row via `await mailTimeInstance.___dispatch(row)` instead of `___send`. `___dispatch` acquires a pool slot and starts the send detached.

## Types

- `MailTimeTask` gains optional `isSending?: boolean` and `sendingAt?: number` fields.
- New `MailTimeIterateOptions` export (`{ limit?: number, sendingTimeout?: number }`).
- `MailTimeOptions` gains `mode?: 'one' | 'batch'`, `concurrency?: number`, `sendingTimeout?: number`.
- `MailTime#drain(): Promise<void>` added to the public surface.
- `CustomQueue#iterate` signature widened to `(opts?: MailTimeIterateOptions) => Promise<void> | void`.
- New `MailTimeRejectedRecipient` export. `MailTimeMailOptions` gains optional `accepted?: string[]` and `rejected?: MailTimeRejectedRecipient[]` fields documenting the in-flight delivery state.
- New `mailTimePreset`, `presets`, and `presetNames` value exports plus `MailTimePresetConfig` and `MailTimePresetName` type exports (via the new `presets.js` module).
- Internal `__` / `___` prefixed members are stripped from generated `.d.ts` / `.d.cts` via `scripts/strip-internal-dts.mjs`.
- Public API types (`MailTimeOptions`, `MailTimeMailOptions`, `MailTimeTask`, `CustomQueue`, etc.) exported and verified by `test/types/{esm.ts,cjs.cts}`.

## Migration

- **Custom adapters** must update their `update` claim guard. The trigger is now `updateObj.isSending === true && typeof updateObj.tries === 'number'` (was `updateObj.isSent === true && ...`). The predicate gains a stale-lock-recovery clause: `(isSending === false OR sendingAt <= now - sendingTimeout)`. See `docs/queue-api.md` and `adapters/blank-example.js`.
- **Custom adapters** must change their iterate path from `await mailTimeInstance.___send(row)` to `await mailTimeInstance.___dispatch(row)` and honor `opts.limit` / `opts.sendingTimeout`.
- Existing rows in production storage have no `isSending` / `sendingAt` fields. Mongo and Redis treat `undefined` as `false` automatically. Postgres `ALTER TABLE ADD COLUMN IF NOT EXISTS` adds them with defaults — no manual migration required.
- Behaviour-wise, default `concurrency: 1` makes the post-upgrade send rate identical to pre-upgrade. Opt into parallel sends by setting `concurrency: N`.

## Migration from 3.x

1. Update Node to ≥ 20.9.0.
2. Replace any adapter import / construction with the new `MongoQueue` / `RedisQueue` / `PostgresQueue` constructors.
3. Pass `josk` config block — it's now required for `type: 'server'`.
4. If you depended on the old `josk.zombieTime` default (`32786`), set it explicitly.
5. Custom queue adapters: `update` now receives claim updates `{ isSending: true, sendingAt: <ms>, tries: N }` (was `{ isSent: true, tries: N }` pre-v4). They must atomically guard on stored `tries`, active state, and the new stale-lock-recovery clause `(isSending === false OR sendingAt <= now - sendingTimeout)`. The iterate path must also call `await mailTimeInstance.___dispatch(row)` instead of `___send`, honoring `opts.limit` and `opts.sendingTimeout`. See [docs/queue-api.md](./queue-api.md) and `adapters/blank-example.js`.
